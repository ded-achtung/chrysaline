# Живая Информация — Living Information

> Парадигма, рождённая в проекте AIOS. Не копия мозга. Не нейросеть. Не LLM. Своё.
> Дата рождения: 2026-03-28
> Статус: Python прототип, 8/8 упражнений, реальный учебник

---

## Одна фраза

**Информация — не данные. Информация — живое существо.**

---

## Проблема которую это решает

Все существующие подходы хранят информацию в КОНТЕЙНЕРАХ:

```
Нейросеть:  веса в матрицах. Данные пассивны.
LLM:        токены в контексте. Статистика.
База данных: записи в таблицах. Мёртвые строки.
Мозг (копия): нейроны + связи. Всё равно контейнер.
AIOS v5.5:   KnowledgeStore + 32 модуля. Завод.
```

Во всех случаях: есть ХРАНИЛИЩЕ и есть ДАННЫЕ. Хранилище обрабатывает данные. Данные сами ничего не делают.

**Живая Информация переворачивает это.** Нет хранилища. Нет контейнера. Есть только существа. Каждый кусок информации — существо которое рождается, ест, растёт, сливается с другими, скрещивается, конкурирует и умирает.

## Пять актов жизни информации

### 1. РОЖДЕНИЕ

Система видит слово впервые → рождается существо.

```
Текст: "Кот ел рыбу"

Рождаются три существа:
  "кот"   — новорождённый, энергия 1.0
  "ел"    — новорождённый, энергия 1.0
  "рыбу"  — новорождённый, энергия 1.0
```

Существо не "записывается". Оно ПОЯВЛЯЕТСЯ. Как организм в среде.

### 2. ПИТАНИЕ

Система видит слово повторно → существо крепнет.

```
Текст: "Кот спал"

  "кот" — УЖЕ СУЩЕСТВУЕТ. Не рождается заново.
  Получает питание. Энергия растёт. Становится сильнее.
  
  "спал" — новорождённый.
```

Часто встречающееся = сильное. Редкое = слабое. Не встречается = умирает.

### 3. СЛИЯНИЕ (merge)

Существа рядом сливаются в ОРГАНИЗМ.

```
"кот" + "ел" → организм "кот·ел"
"кот" + "ел" + "рыбу" → организм "кот·ел·рыбу"

Организм — новое существо. Сложнее. Из частей.
Части знают о ребёнке. Ребёнок знает о родителях.
СЕМЬЯ.
```

### 4. СКРЕЩИВАНИЕ (crossbreed)

Два ПОХОЖИХ организма → абстрактный вид со СЛОТАМИ.

```
"кот·ел·рыбу" × "кот·ел·мышку"

Общее:   "кот", "ел"
Разное:  "рыбу" vs "мышку"

→ Рождается: "кот·ел·$0"
  Слот $0 = {рыбу, мышку}
  
Это АБСТРАКЦИЯ. Она знает: после "кот ел" бывает РАЗНОЕ.
Не записанное правило. Существо которое так устроено.
```

Скрещивание — главный механизм обобщения. Из конкретного вырастает абстрактное. Без программирования. Из жизни.

### 5. СМЕРТЬ

Не встречается → голодает → умирает.

```
Существо "вчера" встретилось 1 раз. Больше не встречается.
Каждый тик: энергия -= 0.01.
Через 100 тиков: энергия = 0 → смерть.

Существо "кот" встречается часто. Энергия = 8.0.
Оно ЖИВЁТ. Потому что ИСПОЛЬЗУЕТСЯ.
```

Знание живёт потому что работает. Как буквы алфавита — ты помнишь их в 30 лет потому что пишешь каждый день.

## Шестой акт: VISITING (хождение в гости)

Существа ходят к родственникам. Это и есть мышление.

```
Вопрос: "Какие гласные?"

Существо "гласные" идёт в гости:
  → к ребёнку "гласные·звуки·это·а" → видит "а"
  → к ребёнку "гласные·звуки·это·о" → видит "о"
  → к ребёнку "гласные·звуки·это·у" → видит "у"
  
Собирает: {а, о, у, ы, и, э}

Не поиск в базе. Не SQL запрос. Не forward pass.
Существо САМО ХОДИТ к тем с кем связано семейными узами.
```

### Visiting решает упражнения

```
"ЖИ или ЖЫ?"
  "жи" идёт в гости → братья: "пишется", "с", "буквой", "и"
  → Ответ: ЖИ (пишется с буквой и)

"Какая часть речи КОШКА?"
  "существительное" → братья: "обозначает", "предмет"
  → Кошка = предмет → существительное

"Как написать Москва?"
  "города" → братья: "пишутся", "с", "большой", "буквы"
  → Города пишутся с большой буквы → Москва
```

## Седьмой акт: ГЕНЕРАЦИЯ

Абстракции разворачиваются в конкретные предложения.

```
Существо "кот·ел·$0" с $0={рыбу, мышку, сметану}

Генерация → подставить каждый вариант:
  "кот ел рыбу"
  "кот ел мышку"  
  "кот ел сметану"

Система не просто ХРАНИТ. Она СОЗДАЁТ.
Из одной абстракции — семейство конкретных фраз.
```

## Восьмой акт: ВОПРОСЫ (ask)

Система отвечает на произвольные вопросы через visiting.

```
Вопрос: "что ест кот?"

1. Слова вопроса: "что", "ест", "кот"
2. "ест" неизвестно — отбросить
3. "кот" — известен → visiting
4. Находит: кот·ел·$0 → $0 = {рыбу, мышку, сметану}
5. Ответ: рыбу, мышку, сметану

Не SQL. Не поиск по ключевым словам.
Существо "кот" САМО пошло к родственникам и принесло ответ.
```

---

### Знание = структура родства

Не записи в таблице. Не веса в матрице. А КТО С КЕМ СВЯЗАН СЕМЬЁЙ.

```
"кот" знает "ел" потому что они вместе ПОРОДИЛИ "кот·ел".
"рыбу" и "мышку" заменяемы потому что их общий РОДИТЕЛЬ — "кот·ел·$0".
"гласные" знает {а,о,у,ы,и,э} потому что это её ДЕТИ.
```

### Обобщение = скрещивание

Не алгоритм кластеризации. Не градиентный спуск. Два похожих организма встретились → родился абстрактный вид.

```
"мама·мыла·раму" × "мама·мыла·окно" → "мама·мыла·$0" {раму, окно}
"кот·был·белый" × "кот·был·пушистый" → "кот·был·$0" {белый, пушистый}
"собака·спала·на·коврике" × "собака·спала·на·улице" → "собака·спала·на·$0" {коврике, улице}
```

### Забывание = голод

Не garbage collection. Не TTL. Существо ГОЛОДАЕТ если им не пользуются. Важное используется часто → живёт. Мусор не используется → умирает. Естественный отбор информации.

### Новое поглощается

Новое слово попадает в существующий слот АВТОМАТИЧЕСКИ:

```
Система знает: "кот·ел·$0" = {рыбу, мышку, сметану}
Приходит: "кот ел колбасу"
→ "колбасу" поглощается в слот $0
→ Теперь: {рыбу, мышку, сметану, колбасу}

Не нужно переобучение. Не нужен новый цикл. 
Существующий вид САМ поглотил новый пример.
```

## Чем это отличается от всего существующего

```
LLM:
  Знает: статистику токенов в 2 триллионах слов
  Не знает: почему. Нет структуры. Нет семей. Нет родства.
  Хранит: матрицы весов. Пассивные числа.
  
Нейросеть (STDP, SNN):
  Знает: связи между нейронами
  Не знает: что эти связи ЗНАЧАТ. Нейрон не ходит в гости.
  Хранит: веса синапсов. Контейнер с числами.

Граф знаний (KG):
  Знает: тройки (субъект, предикат, объект)
  Не знает: контекста. Записи мёртвые. Не растут. Не умирают.
  Хранит: записи в базе. Статичные.

Живая информация:
  Знает: семейные связи. Кто кого породил. Кто с кем похож.
  Понимает: через visiting. Идёт к родственникам, собирает ответ.
  Хранит: НИЧЕГО. Существа сами живут. Нет контейнера.
  Растёт: скрещивание → абстракция. Без программирования.
  Забывает: голод → смерть. Без garbage collection.
  Поглощает: новый пример → в существующий слот. Без переобучения.
```

## Карта границ (что работает, что сломано, что не проверено)

```
РАБОТАЕТ:
  ✓ Категории в любом домене (язык, природа, математика)
  ✓ Иерархия (кот→млекопитающее→животное, 3 уровня)
  ✓ Поглощение нового (дельфин, кит → млекопитающие мгновенно)
  ✓ Кросс-доменный перенос (кот = млекопитающее И существительное)
  ✓ Не-шаблонные ответы ("кто ест мясо?" → {кот, собака})
  ✓ Генерация (кот·ел·$0 → конкретные предложения)
  ✓ Объяснения ("почему кит млекопитающее?" → кормит детёнышей молоком)
  ✓ Масштаб 823 существа (7 сказок + учебник + природоведение)
  ✓ Факты математики (2+3=5 как организм)
  ✓ Математические категории (чётные, нечётные, операции)
  ✓ Код движка НЕ МЕНЯЛСЯ между доменами

СЛОМАНО:
  ✗ ОТРИЦАНИЕ — "страус НЕ летает" → страус становится братом "летает"
    Нет знака связи. "не" = просто слово. Нужна валентность.
  ✗ ЛОКАЛЬНОСТЬ — при большом корпусе "это" ломает query_category
    Нужен вес по "служебности" (авто, не стоп-лист)
  ✗ ИЗБЫТОЧНОЕ РАСЩЕПЛЕНИЕ — "коса" даёт 6 смыслов вместо 3
    Нужен min_cluster_size и вес по редкости контекста
  ✗ ВЫЧИСЛЕНИЕ — 4+4=? → тишина. Слоты $0,$1,$2 независимы.
    Нет функциональной связи между слотами.

НЕ ПРОВЕРЕНО:
  ? Масштаб 10000+ предложений
  ? Многоязычность (английский без изменений)
  ? Самостоятельное обучение (система САМА задаёт вопросы)
  ? Длинные правила (15+ слов)
  ? Грамматические преобразования (падежи, спряжения)
```

### Два класса знаний

```
КЛАСС 1: Декларативные знания (РАБОТАЕТ)
  Категории, факты, иерархии, перечисления, ассоциации.
  Механизм: рождение → слияние → скрещивание → visiting.
  
КЛАСС 2: Функциональные знания (НЕ РАБОТАЕТ)
  Вычисления, преобразования, причинные связи.
  Нужен: новый тип существа — функциональная связь между слотами.
  $0 + $1 → $2, где $2 ЗАВИСИТ от $0 и $1.
  НЕ делать сейчас. Сначала добить Класс 1.
```

### Авто-определение служебных слов (НЕ стоп-лист)

```
ПРИНЦИП: система САМА узнаёт что "из", "в", "на" — служебные.
Не через список. Через статистику жизни.

"кот":  5 уникальных братьев / 15 кормлений = 0.33 → содержательное
"из": 150 уникальных братьев / 200 кормлений = 0.75 → служебное

Метрика: service_score = unique_siblings / times_fed
Высокий score = слишком общее для конкретных выводов.
Visiting и query_category снижают вес, НЕ исключают.

Никаких стоп-листов. Работает на любом языке.
Свойства существа определяются его ЖИЗНЬЮ.
```

### Порядок следующих шагов

```
1. ОТРИЦАНИЕ — валентность связи (positive/negative)
   Без этого все ответы семантически загрязнены.

2. ЛОКАЛЬНОСТЬ — авто-вес служебных через service_score
   Без этого масштаб ломает query_category.

3. EXPLAIN() — явная цепочка от вопроса к ответу
   У системы уже есть зачатки, нужно оформить.

4. ОМОНИМИЯ — min_cluster_size + вес по редкости
   Не сломано, но слишком шумно.

5. ФУНКЦИОНАЛЬНЫЕ СУЩЕСТВА — только после 1-4
   Математика = следующий класс знаний.
```

## Фундамент и производные

Не всё равноправно. Есть ядро, есть режимы поверх.

```
ФУНДАМЕНТ (ядро, без этого ничего не работает):
  1. Существо как единица знания (Creature)
  2. Энергия / питание / голод / смерть
  3. Родитель-ребёнок структура (parent_ids, children)
  4. Слияние (merge) — части → организм
  5. Скрещивание (crossbreed) — похожие → вид со слотами
  6. Поглощение (absorption) — новое → в существующий слот

ПРОИЗВОДНЫЕ РЕЖИМЫ (полезны но не фундамент):
  - Visiting — режим чтения структуры родства
  - Ask — режим ответа на вопросы через visiting
  - Generation — режим создания фраз из абстракций
  - Conflict split — режим расщепления при несовместимых контекстах
  - Teacher dialog — режим внешнего влияния на энергию

Производные режимы НЕ должны становиться самостоятельными модулями.
Они работают ЧЕРЕЗ фундамент, не РЯДОМ с ним.
```

## Честные формулировки

```
СИЛЬНЫЕ НО НЕТОЧНЫЕ:         ЧЕСТНЕЕ:
"нет хранилища"          →   "хранилище = среда жизни существ, не пассивный контейнер"
"мышление = visiting"    →   "visiting = один из базовых режимов мышления"  
"понимание = visiting"   →   "visiting даёт категории и связки, не всё понимание"
"существа сами живут"    →   "существа живут по правилам экосистемы в runtime"
```

## Критерии провала

Если любой из этих пунктов станет реальностью — идея в опасности.

```
ПРОВАЛ 1: Взрыв без пользы
  Корпус растёт → число существ растёт → но полезных абстракций НЕ растёт.
  Метрика: abstractions / creatures < 5% при > 5000 существ.

ПРОВАЛ 2: Структурный мусор в ответах
  Visiting начинает тащить "звуки", "это", "пишется" вместо ответов.
  Метрика: > 30% ответов ask() содержат структурные слова.

ПРОВАЛ 3: Смешение категорий при поглощении
  Absorption смешивает гласные с согласными, действия с предметами.
  Метрика: false absorption rate > 10%.

ПРОВАЛ 4: Расщепление в шум
  Conflict split создаёт 10+ "смыслов" где реально 2-3.
  Метрика: avg clusters per ambiguous word > 5.

ПРОВАЛ 5: Шаблонность ответов
  Ask работает ТОЛЬКО на фразах близких к обучению.
  Новая формулировка того же вопроса → пустой ответ.
  Метрика: < 50% ответов на перефразированные вопросы.

ПРОВАЛ 6: Новый завод
  Появляется "visiting engine", "absorption controller",
  "ecosystem balancer" — отдельные модули поверх существ.
  Метрика: количество строк кода ВНЕ Creature/World > 30%.
```

## Правила проекта

```
ПРАВИЛО 1: Информация живая
  Не записываем данные. Рождаем существа.
  Хранилище = среда жизни, не пассивный контейнер.

ПРАВИЛО 2: Функции из жизни
  Обучение = рождение + питание.
```
