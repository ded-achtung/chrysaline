## Открытые проблемы

### ПРИОРИТЕТ 1: Исключения внутри правил

```
Проблема:
  "птицы летают" — общее правило
  "страус не летает" — исключение
  "пингвин не летает" — исключение
  
  Может ли система ОДНОВРЕМЕННО держать:
    общее правило (птицы → летают)
    И исключения (страус → НЕ летает)?
  
  Не проверено. Отрицание работает для отдельных фактов.
  Но работает ли внутри категории — неизвестно.

Тест:
  Подать: "птицы летают", "воробей летает", "орёл летает",
          "страус не летает", "пингвин не летает"
  Спросить: "кто летает?" → {воробей, орёл} БЕЗ страуса и пингвина
  Спросить: "страус летает?" → НЕТ
  Спросить: "страус это птица?" → ДА
  
  Сложность: страус ДОЛЖЕН оставаться птицей, но НЕ летать.
  Отрицание не должно выкинуть страуса из категории "птицы".
```

### ПРИОРИТЕТ 2: Авто-определение служебных слов

```
Проблема:
  При большом корпусе "это", "в", "на", "с" ломают query_category.
  Они встречаются ВЕЗДЕ → попадают в ответы как мусор.
  
  Стоп-лист = хардкод = нарушение TABULA RASA.

Решение (уже придумано, не реализовано):
  service_score = unique_siblings / times_fed
  
  "кот":  5 уникальных братьев / 15 кормлений = 0.33 → содержательное
  "из": 150 уникальных братьев / 200 кормлений = 0.75 → служебное
  
  Высокий score → снизить вес при visiting/query, не исключать.
  Работает на любом языке без списков.

Тест:
  Подать смешанный корпус (800+ существ).
  query_category("гласные") НЕ должен содержать "это", "в", "на".
  БЕЗ стоп-листа.
```

### ПРИОРИТЕТ 3: Explain() — явные цепочки

```
Проблема:
  ask() даёт ОТВЕТ: "кит = млекопитающее"
  Но не объясняет ПОЧЕМУ.
  
  visit() находит братьев: "кормит", "детёнышей", "молоком"
  Это ЧАСТИ объяснения, но не ЦЕПОЧКА.

Решение:
  explain(A, B) → цепочка организмов от A до B:
    кит → кормит·детёнышей·молоком → млекопитающее
    страус → птица (pos) → летает (neg маркер "не")
  
  Должен работать и с positive и с negative связями:
    explain("почему страус не летает?") →
      страус·не·летает (организм с neg-маркером)
    explain("почему Москва с большой буквы?") →
      москва → город → города·пишутся·с·большой·буквы

Тест:
  explain("кит", "млекопитающее") → путь через общих братьев
  explain("страус", "не летает") → путь через neg-маркер
  explain("Москва", "большая буква") → путь через правило городов
```

### ПРИОРИТЕТ 4: Омонимия — чистое расщепление

```
Проблема:
  "коса" расщепляется на 6 смыслов вместо 3.
  "ключ" на 4 вместо 2.
  Служебные слова ("из") получают 9 мусорных смыслов.

Причина:
  - Нет минимального размера кластера
  - Высокочастотные слова создают ложные пересечения
  - Нет веса по редкости контекста

Решение:
  1. min_cluster_size = 2 (меньше 2 контекстов ≠ отдельный смысл)
  2. Вес контекстного слова = 1/frequency (редкое важнее частого)
  3. service_score > 0.7 → не расщеплять (это служебное слово)

Тест:
  "коса" → 3 смысла (волосы, инструмент, песчаная)
  "ключ" → 2 смысла (дверной, водный)
  "из" → НЕ расщепляется (служебное)
```

### ПРИОРИТЕТ 5: Авто-синонимы (вариант 2 из v18)

```
Проблема:
  Учитель дал "плюс = прибавить" → цепочка заработала.
  Но система могла бы найти это САМА.

Идея:
  "плюс" и "прибавить" стоят в ОДИНАКОВЫХ позициях:
    $0·один → $0 = {плюс, прибавить}
  Если скрещивание обнаружит это — синоним найден автоматически.

Условие:
  Нужно достаточно данных где оба слова в похожих контекстах.
  "один плюс два" + "прибавить один к двум" → 
  скрещивание → $0·один → {плюс, прибавить}

Тест:
  Подать тексты где "плюс" и "прибавить" в похожих позициях.
  Без учителя. Проверить: найдёт ли авто-синоним?

Статус: усиливает учителя, не заменяет.
  Учитель = быстро и точно.
  Авто = медленно но масштабируется на любой язык.
```

### ПРИОРИТЕТ 6: Многоязычность

```
Проблема:
  Вся работа на русском. Работает ли на английском?
  Движок НЕ знает про языки. Но парсер — знает (стемминг, чанки).

Тест:
  Подать: "cats eat fish", "cats eat mice", "dogs eat bones"
  Проверить: кот·ел·$0 → аналог на английском?
  Подать: "vowels are a, e, i, o, u"
  Проверить: query_category("vowels") → {a,e,i,o,u}?

Ожидание: движок работает, парсер нужно адаптировать.
```

### ПРИОРИТЕТ 7: Самостоятельное обучение

```
Проблема:
  Система отвечает на вопросы. Но не задаёт их.
  Не замечает пробелы в знаниях.
  Не говорит "я не знаю X".

Идея:
  Если visiting по слову X возвращает ПУСТО или МАЛО братьев →
  это ПРОБЕЛ. Система может спросить учителя:
    "что такое X?"
    "X это какая категория?"

Тест:
  Подать текст с новым словом "квазар".
  Система не знает что это.
  Должна: обнаружить пробел → сформулировать вопрос.
```

### ПРИОРИТЕТ 8: Длинные правила (15+ слов)

```
Проблема:
  Правило "при произношении гласных звуков воздух проходит
  во рту свободно" = 9 слов.
  Парсер режет на чанки по 5 слов.
  Части правила живут в РАЗНЫХ организмах.

Решение:
  Организм из 9 слов? Слишком длинный, мусорные скрещивания.
  Лучше: ключевые слова правила как БРАТЬЯ через общий организм.
  "гласные·воздух·свободно" → тройка ключевых слов.
  Visiting найдёт связь через эту тройку.

Тест:
  Подать длинное правило (15 слов).
  Спросить: visiting находит ВСЕ ключевые части?
```

## Принцип решения проблем

```
Каждая "невозможная" проблема до сих пор решалась МАЛЕНЬКИМ шагом:

  "Категории не работают"    → скрещивание (одна функция)
  "Отрицание сломано"        → авто-детекция из противоречий
  "Масштаб ломает"           → абстракции переживают конкретику
  "Вычисление невозможно"    → одна фраза учителя

  НЕ БЫЛО ни одного случая где понадобился НОВЫЙ МОДУЛЬ.
  Каждый раз: маленький шаг в существующем фундаменте.

  Если следующая проблема потребует нового модуля — СТОП.
  Сначала подумать: можно ли решить через существа, родство, visiting?
```
